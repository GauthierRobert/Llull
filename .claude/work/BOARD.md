# llull build board — single source of truth for "continue working"

This file is **state**. The `continue-working` skill reads it, reconciles it against the
actual repo, launches parallel agents for the next eligible tasks, and writes it back.
The user only ever has to say **"continue working"**. Humans may edit this file directly.

## Status tokens (line-start, machine-parsed)
- `[TODO]` not started · `[WIP]` an agent is on it now · `[REVIEW]` code done, awaiting `cad-reviewer` + `npm run check`
- `[DONE]` merged & green · `[BLOCKED]` needs a decision/dependency (see note) · `[REMOVED]` deliverable deleted by a later decision (see note)

## Conventions
- **Deps** reference other task IDs. A task is *eligible* only when every dep is `[DONE]`.
- **Lanes** own disjoint file sets (below) so parallel agents don't collide. One task per
  lane runs at a time. Default parallelism = **4 lanes** (+ Lane 5 review runs after each).
- The only shared file is `src/core/commands/registry.ts` — **Lane 1 owns it**; no other
  lane edits it.

## How parallel agents coordinate (they don't talk — they don't need to)
Agents are isolated; coordination is structural, run by the orchestrator:
1. Only the orchestrator writes this board; a task is flipped to `[WIP]` **before** its agent
   launches → the board is the live "who's on what" ledger. Agents never edit it.
2. Disjoint lane file-ownership means two agents **cannot** write the same file.
3. A `[WIP]` lane is busy → never re-assigned while in flight.
4. Cross-lane needs wait on `[DONE]` deps and read the result from **committed files** — the
   repo is the only channel between lanes.
5. `[REVIEW]` (cad-reviewer + `npm run check`, + merge if worktrees) is the backstop.

# WAVE 3 — "from capabilities to a product"

Wave 1 built the foundation; Wave 2 expands capability (parametric, measure, layers, export, modeling
depth, AI/MCP tools). **Wave 3 hardens that foundation, adds the product-level features that make llull a
*real* CAD app (assemblies, drawings, materials, B-rep), and deepens the highest-leverage existing
surfaces (MCP/AI, parametric, measurement, viewport UX).**

Lane file-ownership and the parallel protocol are **unchanged** (see top of file). Many Wave 3 tasks dep on
Wave 2 deliverables — deps are stated honestly; eligibility is still gated by every dep being `[DONE]`.

**Headline themes (twin pillars):**
1. **"This is real CAD now"** — Assemblies + Drawings on a true B-rep kernel (NF1–NF3, NF5, KI4).
2. **"The best agent-driven CAD"** — deepen the MCP/AI surface: session model, structured data, prompts,
   tool annotations (KI1, KI2, EN1–EN4).

**Sequencing:** front-load the foundation (KI1–KI3, KI5) because the feature work deps on it; then run the
two pillars in parallel across the lanes. Collaboration / simulation / CAM stay "Later" (roadmap) — out of
Wave 3 scope to keep the wave shippable.

# WAVE 4 — reliable agent-driven 3D modeling

**Premise (user-reported):** an MCP agent (Claude) repeatedly *failed* to create good 3D objects. Root cause is **NOT** delivery — the MCP→UI live path works (`server/src/liveDocument.ts` `setLiveDoc()` → SSE `GET /live` → `src/ui/hooks/useMcpLiveDocument.ts` `hydrateLiveDocument()`; the box DOES appear live). The cause is that **the agent produces wrong/misplaced geometry and cannot tell.** Wave 4 closes that gap across three pillars: **(1) tool capability**, **(2) the agent's context** (what a blind agent reads), **(3) the feedback loop** (what the agent SEES). Lane file-ownership + the parallel protocol are unchanged.

**Diagnosis (the four friction points):**
1. **Placement ambiguous + inconsistent** — `add_box` "position" is the *center* but undocumented (`geometry.ts:30-67`); `add_cylinder` says "base center" yet bounds are ±height/2; cone/pyramid/wedge each anchor differently. The agent can't predict where geometry lands.
2. **No rotation at creation** — every `add_*` hardcodes `rotation:[0,0,0]`, no `rotation` param → can't orient a part in one call. (Viewport already renders `entity.rotation` extrinsic-XYZ, so storing it on creation needs no Lane-2 work.)
3. **Frame-blind vision** — `render_view` returns SVG/PNG with no world axes, no grid, no units/scale label; bounds are AABB (rotation ignored). The agent can't judge size/placement.
4. **No agent-facing conventions over MCP** — units, +Z-up RH frame, the anchor convention, and the `describe_scene → create → render_view → check_model` loop live only in code; EN2 prompts emit plans but there is no conventions resource an agent reads first.
